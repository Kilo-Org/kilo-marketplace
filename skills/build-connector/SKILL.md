# Build a Fivetran Connector SDK Connector

This package is self-contained. It does not include SDK reference files, workflow subagents, or credential helper scripts. Use the installed `fivetran` CLI and the official Connector SDK documentation at <https://fivetran.com/docs/connector-sdk> for version-specific behavior.

## 1. Check for a Simpler Managed Path

Before generating code:

1. Search <https://fivetran.com/docs/connectors> and <https://fivetran.com/integrations> for the source.
2. If a managed connector exists, explain that it avoids custom connector maintenance.
3. For a SaaS REST API with JSON responses and standard API-key, bearer-token, or OAuth 2 authentication, mention the managed Lite Connector option: <https://fivetran.com/docs/connectors/applications/lite-connectors>.
4. Continue with a custom Connector SDK connector only when the user needs custom behavior or explicitly chooses it.

Do not block progress on catalog certainty. Record what was checked and label uncertain matches as candidates rather than exact matches.

## 2. Validate Requirements

Build a specification before editing code. Capture:

- Source name, API base URL, and authoritative API documentation
- Authentication type and required configuration field names; never request secret values in chat
- Objects/endpoints to sync and the expected destination table names
- Primary keys, update cursors, deletion behavior, and historical backfill requirements
- Pagination, rate limits, retries, timeouts, and incremental-state semantics
- Nested-object normalization and data-type mappings
- Required and optional configuration fields
- A small, non-sensitive test scope

Classify the starting point as:

- `EXACT MATCH`: an installed/community template targets the same source
- `FUZZY MATCH`: a template has a closely related API or authentication pattern
- `BUILD ON TEMPLATE`: no suitable source-specific template exists

Inspect available CLI syntax and templates instead of guessing:

```bash
fivetran --help
fivetran init --help
```

If critical API behavior is ambiguous, ask only the focused questions needed to settle the specification.

## 3. Scaffold

Use the CLI scaffold rather than hand-writing a project. The connector directory should use lowercase letters and underscores.

For a verified community template:

```bash
printf '\n' | fivetran init "<connector_dir>" --template "connectors/<template_name>" --force
```

For the default template:

```bash
printf '\n' | fivetran init "<connector_dir>" --force
```

On PowerShell, replace `printf '\n' |` with `"" |`. Verify that `<connector_dir>/connector.py` exists after initialization. The piped newline only bypasses optional agent-integration setup; do not infer success solely from the process exit code.

## 4. Customize the Scaffold

Edit the generated files in place and preserve the scaffold's supported structure, including `validate_configuration()`, module-level `Connector(...)`, docstrings, and the `__main__` block when present.

### `configuration.json`

- Define only fields required by the specification.
- Distinguish required and optional fields.
- Use descriptions and placeholders, never real credentials.
- Keep secrets out of source control.

### `connector.py`

- Validate configuration before making requests.
- Use bounded timeouts and retry only transient failures.
- Implement pagination termination explicitly.
- Persist incremental state only after the corresponding records are yielded successfully.
- Use stable primary keys and deterministic table names.
- Avoid logging tokens, credentials, full authorization headers, or sensitive payloads.
- Handle missing optional fields and malformed records without corrupting state.

### `README.md`

Document supported objects, configuration field names, incremental behavior, known limits, and safe local test instructions. Do not claim support for endpoints that were not implemented.

## 5. Set Up the Environment

Use the dependency file generated by the scaffold.

macOS/Linux:

```bash
cd "<connector_dir>"
uv venv .venv
uv pip install --python .venv/bin/python -r requirements.txt
```

Windows PowerShell:

```powershell
cd "<connector_dir>"
uv venv .venv
uv pip install --python .\.venv\Scripts\python.exe -r requirements.txt
```

Install `fivetran_connector_sdk` separately only if the generated dependency file does not already include it.

## 6. Configure and Test Safely

This package does not ship a configuration-entry or secure-runner helper. Do not claim otherwise.

- Never ask the user to paste credentials into chat.
- Have the user populate configuration through the mechanism supported by their installed CLI/project, discovered with `fivetran --help` and the generated README.
- Do not read, print, commit, or echo secret-bearing configuration.
- Start with a small test scope and a non-production account when available.

Before an end-to-end run, perform local checks that do not require secrets:

```bash
python -m compileall connector.py
```

Then use the test/debug command exposed by the installed `fivetran` CLI. Inspect its help first because command names and flags can vary by SDK release:

```bash
fivetran --help
```

Report the command used, tables produced, row counts, state changes, and any redacted errors.

## 7. Diagnose Failures

Classify before changing code:

- `INFRA`: DNS, TLS, network, runtime, or SDK installation failure. Fix the environment, not connector logic.
- `CONFIG`: missing/invalid credentials, permissions, tenant, or source settings. Ask the user to verify locally without revealing values.
- `API`: undocumented response shape, rate limit, pagination, or source-side error. Re-check authoritative API docs and captured responses with secrets redacted.
- `CODE`: syntax, mapping, state, or SDK misuse. Make the smallest code fix and rerun static checks plus the narrow test.

For code failures, verify pagination termination, state advancement, null handling, primary keys, retries, and table naming. Never modify the installed CLI or SDK package to hide a connector bug.
